Integration options

Four realistic patterns.

These are not abstract technical ideas. They are actual delivery options with different trade-offs in scope, UX continuity, API dependency, cost, and implementation risk.
Option 1

Direct redirect to app

The website explains the product and sends all high-intent actions directly to app.trainsit.com. No quote search or results are shown on the website itself.
Fastest / leanest
How it works
1User lands on Home / Solutions / About.
2User clicks Get a Quote or Login.
3Website redirects immediately to app.trainsit.com.
4App handles auth, quote flow, dashboard, and every product interaction.
What it requires
ANo website-side API integration.
BStable app URLs and predictable redirect paths.
CClear app entry handling for logged-in and logged-out users.
Very low implementation risk Minimal dependency on internal API work Weakest marketing-to-product continuity User leaves website too early
ProsConsBest fit
Fastest to deliver. Lowest integration complexity. Strong separation between site and app. Website feels like a brochure. No route search preview. Users hit the app before trust is fully built. Useful only if Phase 1 must be extremely compressed or the internal API is not ready.
Option 2

Website-side “Get a Quote” form + API results + redirect to app

The website includes a quote/search form, calls the existing API, displays result states on the website, and sends users to the app only when they want to continue into booking, login, or account-level workflows.
Best Phase 1 fit
How it works
1User clicks Get a Quote on the website.
2User fills origin, destination, date, and optional fields.
3Website sends JS request to internal route/quote API.
4Website shows result / no-result / error state inside the page.
5User clicks Book, Continue, or Login and is sent to app.trainsit.com.

Option 3

Application Gateway page

Instead of sending users straight into the app or only showing a quote form, the website sends them to a dedicated gateway page that decides the right next action based on intent: quote, login, continue, or fallback contact/support.
Future-proof routing layer
How it works
1User clicks CTA from a marketing page.
2User lands on /start or /application-gateway.
3Gateway asks or infers intent: search quote, login, continue, or contact team.
4Gateway routes user into the correct next flow.
What it requires
AWebsite page and UI logic for routing states.
BPotential session-awareness or query-param rules.
COptional API call if quote/search is also embedded into the gateway.
Best long-term control of handoff Good analytics and routing visibility More scope than Option 2 Needs sharper UX definition
ProsConsBest fit
Creates a clean website-to-app orchestration layer. Stronger long-term flexibility if multiple app entry paths are expected. More UX and logic design. Can become overengineered if the real Phase 1 need is only one quote/search flow. Best if Trainsit wants a future-ready access model beyond a single quote form, or wants to differentiate new users, returning users, and evaluator flows.

Recommendation

Recommended approach.

Recommendation: Option 2 as the Phase 1 baseline, with a light Option 3 upgrade path.

The most rational Phase 1 choice is to implement a website-side Get a Quote experience that calls the internal API, displays result states on the website, and sends users to the app only when they need account-level actions. This matches the brief’s intent that the website be both a marketing layer and a platform gateway, while still keeping the website lean and within the 30–40 day target. The architecture can be designed so that a future Application Gateway layer can be added later without rethinking the whole website. fileciteturn5file3turn5file6turn5file7

Why this is the best fit

It gives users immediate value, keeps evaluators inside a trust-building website context longer, supports platform visibility, and avoids turning the website into a second product codebase. It also matches the previously defined markup that places Get a Quote as the only real API integration in Phase 1. fileciteturn5file4turn5file5

What must be true for it to work

The internal team must provide the endpoint, method, parameters, auth rules, response contracts, CORS, and redirect logic early. Without that, Option 2 degrades into Option 1 by necessity. fileciteturn5file0turn5file2

Phased rollout
1Phase 1: Website + Get a Quote form + API results + redirect to app.
2Phase 1.5: Add a light Application Gateway only if multiple entry scenarios create confusion.
3Phase 2: Consider richer routing, session-awareness, or deeper continuity between website and app.
4Later: Revisit embedded app modules only if the business decides the website and app should behave like one product shell.

Main delivery risk

The highest realistic risk is not frontend build speed. It is ambiguity in the app handoff and late API clarification. In BABOK terms, this is an interface-analysis and dependency-definition problem, not just a design problem. Interface analysis is explicitly about identifying who exchanges what information, where, when, why, and how across solution boundaries. fileciteturn5file13

Methodological note

A hybrid approach is appropriate here: the website structure and page build can be designed predictively, while the integration layer should stay adaptive until the API contract and redirect behavior are confirmed. That matches PMBOK guidance on hybrid delivery when uncertainty is concentrated in one stream while the rest of the solution can be specified more predictively. fileciteturn5file8

This page is intentionally decision-oriented: it separates what the website must do, what the app must continue doing, what the interface between them must support, and which option is most defensible under the current scope, risk, and timeline.
